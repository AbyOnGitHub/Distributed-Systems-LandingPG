Distributed Systems & DFS

by- Abedan Biswal and Atharv Ghorpade

"A Distributed System is a collection of independent computers that appears to its users as a single coherent system."
— Andrew Tanenbaum

Introduction to Distributed File Systems (DFS):
A DFS is a file system that allows files to be stored across multiple machines (nodes) in a network, while presenting them to users as if they exist on a single, unified system. Think of it like Google Drive — your files are stored across hundreds of servers, but you see them in one place.

Characteristics

Concurrency

Multiple users/processes access resources simultaneously.

No shared clock

Nodes have independent clocks; synchronization is hard.

No shared memory

Nodes communicate only via messages over a network.

Fault Tolerance

System continues working despite partial failures.

Scalability

Can grow by adding more nodes without redesign.

Openness

Follows standard protocols; components can be replaced.

Heterogeneity

Works across different OS, hardware, and networks.

Advantages

Resource Sharing & Scalability

Files, printers, and databases are shared across machines. Add nodes to handle more load easily.

Performance & Cost Efficiency

Parallel access and local caching speed things up. Commodity hardware is cheaper than one giant machine.

Geographic Distribution & Transparency

Users in different locations access the same data. Complexity is hidden; users get a simple experience.

Naming & Transparency

What is Naming?

Naming is the mechanism by which resources (files, nodes, services) in a distributed system are identified and located.

Naming Structure

LayerDescriptionExample
Low-level namesMachine-level identifiersinode 4821
Mid-level namesHuman-readable identifiers/home/user/file.txt
High-level namesUser-friendly, symbolic nameswww.example.com

Naming Schemas

1. Flat Naming

Unstructured strings with no hierarchy. Simple but hard to scale (e.g., MAC addresses).

2. Structured Naming

Organized in a tree-like hierarchy. Easy to scale (e.g., DNS, Unix file paths).

3. Attribute-Based

Resources identified by properties rather than a fixed name (e.g., LDAP searches).

Transparency

Hiding the complexity of distribution from users and applications.

TypeMeaning
Access TransparencyLocal and remote resources accessed the same way
Location TransparencyYou don't need to know where a file physically lives
Replication TransparencyMultiple copies exist, but you see just one
Failure TransparencySystem hides crashes and recovers silently
Migration TransparencyFiles can move between servers without user knowing

Implementation Techniques

1. RPC & Communication

Client calls a function that executes remotely (RPC). Nodes communicate via protocols like NFS, SMB, or HDFS protocol.

2. Partitioning & Caching

Data is split across nodes (Sharding). Files are copied locally on the client to reduce network calls.

3. Consistency & Consensus

Algorithms like Paxos or Raft ensure nodes agree on a single truth. Consistency models dictate how versions are synced.

4. Naming & Location

A Name Server or Metadata Server maps file names directly to physical block locations.

5. Security & Fault Handling

Heartbeats signal nodes are alive. Security handles Authentication, Authorization, and Encryption.

6. Consistent Hashing

Efficiently maps data to nodes; minimizes reshuffling when nodes are added or removed.

Real-World Architecture Simulators

HDFS: Block Partitioning & Replication

Hadoop Distributed File System (HDFS) is designed for storing massive datasets across commodity hardware. It works by breaking large files down into standardized blocks (typically 128MB). These blocks are scattered across multiple "DataNodes" and replicated (usually 3 times) to ensure fault tolerance. This architecture allows frameworks like Spark to process chunks of a file in parallel across many machines.

NFS: Remote Directory Mounting

Network File System (NFS) is a classic client/server protocol. Its primary aim is Access Transparency. A client can "mount" a directory that physically lives on a remote server directly onto their local file system tree. To end-users, the remote files behave exactly like local files. Under the hood, the OS intercepts file operations and translates them into Remote Procedure Calls (RPC) sent over the network.

Amazon S3: Flat Object Storage

Unlike HDFS or NFS, S3 is an Object Store, not a traditional file system. There are no actual directories or hierarchical file trees at the physical storage layer. Data is stored as individual "Objects" inside a massive, flat "Bucket". Each object is retrieved via a unique Key. The folders you see in S3 are simply an illusion created by UIs using the `/` character in the key name. This allows infinite horizontal scalability.
